# Release Notes Style Guide

This guide explains how release notes should be written for GatewayOps Cloud.

## Goals

Release notes should:

- Explain what changed
- Tell users why it matters
- Identify required action
- Document known issues honestly
- Separate user-facing details from internal details

## Voice and Tone

Use a clear, direct, helpful tone.

Avoid:

- Engineering jargon
- Vague phrases like "various improvements"
- Overly promotional language
- Blaming users for issues

Use:

- Plain language
- Specific feature names
- Clear action steps
- Honest issue descriptions

## User-Facing Release Notes

User-facing release notes should answer:

1. What changed?
2. Why does it matter?
3. Do I need to do anything?
4. What problems were fixed?
5. Are there any known issues?

## Internal Release Notes

Internal release notes should include:

- Engineering context
- QA status
- Rollout details
- Support impact
- Risk level
- Escalation guidance

## Writing Examples

Weak:

> Improved pairing.

Better:

> The pairing page now shows each stage of the pairing process, so technicians can see where a failed pairing attempt stopped.

Weak:

> Fixed bugs.

Better:

> Fixed an issue where offline gateways sometimes appeared as active.

## Breaking Changes

Breaking changes must include:

- What changed
- Who is affected
- Before-and-after examples
- Required action
- Migration guidance

## Known Issues

Known issues should include:

- The issue
- User impact
- Workaround
- Current status
